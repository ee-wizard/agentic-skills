# 代码实现 Skill

## 概述

本 Skill 用于根据给定的需求描述，生成完整、正确、可维护的代码实现。核心目标是在理解需求的基础上，输出符合项目规范的高质量代码。

## 使用流程

1. **需求理解** — 仔细阅读需求描述，识别关键功能点、边界条件和约束
2. **上下文收集** — 分析项目结构、现有代码风格、已使用的库和框架
3. **方案设计** — 确定实现思路、模块划分、数据结构选择
4. **代码实现** — 编写代码，确保正确性、可读性和一致性
5. **验证确认** — 检查代码是否符合需求、编译/语法是否正确

## 实现原则


### 通用原则

#### 禁止隐式 Fallback

- 用户明确指定的需求（如 GPU 加速、特定库/框架、特定算法）**必须严格遵循**，代码中**不得包含任何隐式 fallback 逻辑**
- 例如：用户指明使用 GPU，则环境不支持 GPU 时应直接报错退出，而非静默回退到 CPU
- 若某些环境可能导致需求无法满足，应在运行入口处做前置检查（`check_environment()`），不满足时抛出清晰的错误信息并终止执行
- 例外情况：用户明确说明 "如果 A 不可用则用 B" 时，可按约定 fallback

### 配置管理（YAML）

- 项目中所有需要外部参数的运行文件，优先使用 **YAML 配置文件** 而非硬编码参数
- 每个可独立运行的文件（脚本、入口点）在 `scripts.yaml` 中独占一个唯一 `ID`，通过 ID 关联其配置
- 整个项目根目录下维护一个统一的 `scripts.yaml`，结构如下：

```yaml
# scripts.yaml — 项目统一配置
id_maps:
  - scipt_1: script_id_1
  - scipt_2: script_id_2

script_id_1:
  description: "脚本功能简述"
  params:
    param_a: value_a
    param_b: value_b

script_id_2:
  description: "另一脚本功能简述"
  params:
    param_x: value_x
```

- 代码中通过 `ID` 加载配置，禁止在脚本内硬编码业务参数
